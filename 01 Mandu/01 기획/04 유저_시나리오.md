1. PM A가 서비스에 로그인한다.
    로그인 후 “내 프로젝트 목록” 화면이 열린다.
2. “내 프로젝트 목록”에서 주요 프로젝트를 선택한다.
    각 프로젝트 카드에는:
     * 프로젝트 이름
     * 전체 Task 진행률(간단)
     * “이번 주 활동량” 요약 (Commit/Task 변경 수)
    PM A는 오늘 체크해야 할 프로젝트를 선택한다.
3. 프로젝트 상세 화면에서 “Flow 보기” 탭을 연다.
    해당 프로젝트에 등록된 Flow 목록이 보인다.
     예: `회원가입 Flow`, `결제 Flow`, `주문 취소 Flow`
    각 Flow 옆에는:
      완료된 Node 비율 (예: 5/8 단계 진행)
      최근 활동 시간(해당 Flow 관련 Task/Commit 기준)
4. PM A가*`결제 Flow`를 선택한다.
    Flow Chart 뷰 또는 Node 리스트 뷰가 열린다.
    각 Node에 대해:
     * Node 이름 (예: “카드 정보 입력”, “결제 승인 요청”, “영수증 발송”)
     * 연결된 Task 수
     * DONE Task 비율
     * 최근 Commit 시간
    PM A는 여기서:
     * Task가 하나도 없는 Node
     * Task는 있지만 대부분 TODO/IN\_PROGRESS인 Node
     * 최근 며칠간 활동이 없는 Node를 한눈에 파악한다.
5. PM A는 “진행이 부족해 보이는 Node” 몇 개를 메모한다.
   예:
     “취소/환불 처리” Node에 Task 1개, IN\_PROGRESS인데 최근 커밋 없음
     “에러 처리/로그 기록” Node에 Task 자체가 없는 상태
   * 이 Node들을 오늘 개발자와 논의할 주요 포인트로 잡는다.


2.4 시나리오 2: Flow 기준으로 “위험 단계” 찾기 (마감 임박 + 활동 없음)

목적
**마감이 가까운 기능 Flow 중**,
  **특정 단계에서 실제 작업이 거의 없는 곳**을 빠르게 찾아낸다.
흐름
1. PM A가 프로젝트의 “Flow 위험 보기” 기능을 연다.
   메뉴 예: `Flow 리스크 보기`
   필터:
     * 기간: 이번 주
     * 조건: “마감 임박” + “최근 활동 없음”
2. 시스템이 Flow Node 기준으로 위험 단계를 계산한다.
   ES에 쌓인 활동 로그를 기준으로:
     Node에 연결된 Task들 중,
       * 마감일까지 N일 이하 남았고
       * 최근 X일 동안 Commit/상태 변경이 없는 Task 비율이 높은 Node를 “위험 Node”로 판단한다.
3. PM A는 위험 Node 리스트를 본다.
   예시 표시:
     `결제 Flow / 환불 처리 단계`
       * 연결 Task: 3개
       * 마감 임박 Task: 2개
       * 최근 3일간 활동 없음
     `회원가입 Flow / 이메일 인증 단계`
       * 연결 Task: 1개
       * 상태: IN\_PROGRESS
       * 최근 커밋: 5일 전
4. 위험 Node를 클릭해 상세 정보를 본다.
   Node 상세 화면:
     * 이 Node에 연결된 Task 리스트
       * 각 Task의 상태, 담당자, 마감일
       * 연결된 Commit 수, 최근 Commit 시간
   * 예:
     * Task A: IN\_PROGRESS, 담당자 B, 마감 이틀 남음, 커밋 0개
     * Task B: TODO, 담당자 없음, 마감 3일 남음
5. PM A가 후속 액션을 결정한다.
   예:
     * 담당자가 없는 Task에 담당자 배정 요청
     * 커밋이 없는 IN\_PROGRESS Task에 대해 진척 상황 문의
     * 마감 조정/우선순위 재조정 논의
6. 이 Flow 위험 화면은 PM의 “리스크 레이더” 역할을 한다.
   * 매일/주기적으로 확인하면서,“어디가 터질지 모르는 구간인지” 를 미리 발견한다.


2.5 시나리오 3: 특정 Flow 단계의 실제 개발 히스토리 확인

목적
특정 Flow 단계에 대해,
  * 어떤 Task들이 연결되어 있고
  * 그 Task들이 어떤 Commit들로 구현되었는지를 한 번에 보고, 기능 단위 히스토리를 이해한다.
흐름
1. PM A가 회의 중 “결제 승인 요청 단계”에 대해 이야기해야 한다.
   * 이 단계가 언제부터 작업이 시작되었고,
   * 어느 정도 커밋/변경이 있었는지 알고 싶다.
2. PM A가 Flow 화면에서 해당 Node를 클릭한다.
   * Node 상세 화면이 열린다.
   * 위쪽: Node 정보 (이름, 설명, 중요도 등)
   * 중간: 연결된 Task 리스트
   * 아래: 해당 Node와 연결된 Task에서 나온 Commit 타임라인
3. Task 리스트 확인
   * Task 목록:
     * Task1: “결제 승인 API 구현”, DONE
     * Task2: “결제 승인 실패 처리”, IN\_PROGRESS
     * Task3: “로그/모니터링 추가”, TODO
   * 각 Task 옆에:
     * 담당자
     * 상태
     * 최근 Commit 시간
4. Commit 타임라인 확인
   시간순으로 정렬된 Commit 목록:
     * 11/10 10:12, B, “결제 승인 API 기본 구현 (PROJ-101)”
     * 11/11 14:03, C, “결제 승인 실패 처리 로직 추가 (PROJ-102)”
     * 11/12 09:20, B, “결제 승인 로그 추가 (PROJ-103)”
   PM A는 이 타임라인만 보고:
     * 언제 작업이 시작되었고
     * 어느 시점에 집중적으로 작업이 이루어졌는지를 이해할 수 있다.
5. 필요하면 Git 상세로 이동
   * 특정 Commit을 클릭하면 외부 Git 서비스의 상세 페이지로 이동
   * PM이 개발자와 함께 코드를 리뷰하거나 설명을 들을 수 있다.
6. PM A는 이 정보를 바탕으로 상위 보고/공유를 한다.
   * 예:
     * “결제 승인 요청 단계는 지난 3일간 집중적으로 개발되어,
         핵심 API는 완료되었고, 아직 실패 처리/로그 작업이 남아 있습니다.”

2.6 시나리오 4: 주간 프로젝트 평가/보고 준비 (Flow + ES 분석 기반, 확장)

목적
* PM이 주간 회고/보고를 준비할 때,
  * 프로젝트별/Flow별 진행 상황과
  * 리스크/이슈를 빠르게 정리할 수 있도록 돕는다.
흐름
1. **PM A가 “주간 프로젝트 평가” 화면을 연다.**
   * 기간 선택: 이번 주 (월\~일)
   * 프로젝트 선택: 특정 프로젝트 또는 전체
2. **시스템이 ES에서 지표/점수를 가져온다.**
   * `project_activity`, `project_score` 같은 인덱스를 기반으로:
     * Task 생성/완료 수
     * 마감 지연 Task 수
     * Flow별 활동량
     * “마감 임박 + 활동 없음” Task/Node의 개수
     * 프로젝트 종합 점수(Delivery/Flow/Risk)
   * 를 한 번에 로딩한다.
3. **프로젝트별 개요를 본다.**
   * 각 프로젝트에 대해:
     * 종합 점수 (예: 72점)
     * 완료 Task 수 / 생성 Task 수
     * 지연 Task 수
     * 위험 Flow/Node 수
   * PM A는 여기서 “이번 주에 신경 더 써야 할 프로젝트”를 고른다.
4. **선택한 프로젝트의 Flow 단위 평가를 본다.**
   * Flow 리스트:
     * `회원가입 Flow` : overall 80점, 지연 Node 0
     * `결제 Flow` : overall 65점, 지연 Node 2
   * 각 Flow를 클릭하면:
     * Node별 지표/상태가 나타난다.
     * 예: `환불 처리` Node에서 지연 Task 다수 발견
5. **(확장) LLM 기반 자연어 요약 보기**
   * PM A가 “주간 요약 생성” 버튼을 누른다.
   * 시스템은 ES 지표를 바탕으로 LLM에 요청한다.
     * 예: “이번 주 결제 프로젝트의 진행 상황과 리스크를 요약해줘”
   * LLM이 생성한 요약:
     * “이번 주 결제 Flow에서는 결제 승인 단계가 계획대로 완료되었지만,
       &#x20;환불 처리 단계에 지연 Task가 2개 존재합니다. 특히, 로그/모니터링 강화 작업이
       &#x20;마감일을 넘길 가능성이 있으므로 우선순위 조정이 필요합니다.”
   * PM A는 이 텍스트를 회의·보고용 자료로 활용하거나, 조금 수정해서 사용한다.

2.7 정리

이 문서에서 정의한 Flow 포함 사용자 시나리오의 핵심은:
* **Flow(설계) – Task(계획) – Commit/활동(실행) – ES 분석/평가**
이 네 가지를 기준으로:
  1. 아침에 Flow 기준으로 전체 진행 상황을 훑어보고
  2. 위험한 Flow 단계/Task를 자동으로 찾아내고
  3. 특정 Flow 단계의 실제 개발 히스토리를 이해하며
  4. 주간 단위로 프로젝트/Flow별 상태를 평가·보고하는 흐름을 PM/기획자가 자연스럽게 사용할 수 있게 만드는 것
