01 서비스 개요
1.1서비스 한 줄 정의

기획/PM이 설계한 흐름(Flow Chart), 관리하는 Task, 실제 코드 커밋과 활동 데이터를 한 곳에 모아서,

 1.2 서비스 목표

1) “설계 → 계획 → 실제 작업”을 한 눈에 잇기

이 서비스는 다음 세 가지를 **하나의 관점으로 묶는 것**을 목표로 한다.

1. **설계 흐름(Flow Chart)**
   * 기획/PM이 기능·서비스의 전체 흐름을 정의한 다이어그램
   * 예: "회원가입 플로우", “결제 플로우”
2. **업무 계획(Task)**
   * Flow의 각 단계별로 쪼개진 작업 단위
   * 상태, 담당자, 마감일 등을 가진다.
3. **실제 실행(Commit·활동 데이터)**
   * 개발자가 Git에 남기는 커밋
   * Task 상태 변경, 작업 활동
이 세 축을 자동/반자동으로 연결해서:
* “이 Flow의 이 단계가 Task로 잘 쪼개져 있는지”
* “Task들이 실제로 커밋과 활동으로 이어지고 있는지”
* “어디 단계에서 지연·병목이 생기는지” 를 **한 화면에서 확인할 수 있도록** 하는 것이 핵심 목표다.

2) PM/기획자가 “업무가 제대로 돌아가고 있는지”를 빨리 판단

이 서비스의 1차 관점은 **개발자가 아니라 PM/기획자**다.
* PM/기획자는 다음을 빠르게 알고 싶어 한다.
  * 이번 주에 어떤 Flow/기능이 실제로 진행되고 있는가?
  * 어느 단계/Task에서 작업이 멈춰 있거나 위험한가?
  * 마감이 가까운 Task 중 실제 코드 작업이 안 된 것은 무엇인가?
이 서비스는:
* Task 상태만 보는 것이 아니라,
* Flow Chart + Task + Commit + 활동 타임라인을 묶어서 보여주고,
* "위험 신호"를 자동으로 골라내는 것에 초점을 둔다.

 3) Elasticsearch 기반 프로젝트 분석/평가

서비스는 **검색/분석 엔진(예: Elasticsearch)** 를 적극적으로 사용한다.
목표:
* 프로젝트별로:
  * Task 생성/완료
  * Commit 수, 활동량
  * 지연 Task, 손 안 대고 있는 단계
같은 지표들을 모아서 **프로젝트 상태를 점수/지표로 평가한다.
이를 통해:
* 프로젝트 상세 화면에서:
  * "이번 주 프로젝트 건강 점수"
  * “Flow 단위로 어느 단계가 위험한지”
같은 정보를 PM이 바로 확인할 수 있게 한다.

1.3 주요 타깃 사용자

 1) PM / 기획자 (1순위)

* 역할
  * 프로젝트 계획 수립
  * Flow 설계
  * Task 분해 / 우선순위 / 마감 관리
  * 프로젝트 진행 상황 모니터링, 리스크 관리
* 이 서비스에서 하고 싶은 일
  * 설계한 Flow Chart 기준으로, 각 단계별 Task/작업 진행 상태를 확인
  * 마감이 다가오는데 작업이 거의 없는 구간을 빠르게 찾기
  * 프로젝트별/기간별 활동 지표와 평가 점수 확인
  * 회의/보고에 사용할 수 있는 주간·월간 요약 관점 확보


2) 팀 리더 / 리드 개발자 (2순위)

* 역할
  * 팀의 개발 일정·우선순위 조정
  * 병목·리스크가 있는 Task를 조기 발견
* 이 서비스에서 하고 싶은 일
  * 특정 Flow/기능의 구현 진행 상황을 Task·Commit 기준으로 확인
  * 담당자별 작업 분포/부하 확인

3) 개발자 (3순위)

* 역할
  * Task를 처리하고 Git에 커밋을 남김
* 이 서비스에서 하는 일
  * 자신에게 할당된 Task 확인
  * 커밋 메시지에 Task 번호를 포함해 Task–Commit이 자동으로 연결되게 사용
  * Flow/Task 단위로 자신이 어떤 부분을 담당하고 있는지 확인

1.4 해결하고자 하는 문제

기존 상황

* 기획/PM은:
  * 문서/Flow Chart로 기능 흐름을 설계하고
  * 별도의 이슈 트래커로 Task를 관리하고
  * 또 다른 도구(또는 Git 웹 UI)로 Commit을 보는 식으로,
  * **정보가 여러 시스템에 흩어져 있고 연결이 약하다.**
* 그래서 반복되는 질문:
  * "이 Flow의 이 단계는 지금 어디까지 됐나요?"
  * “이 기능(Task)은 실제 코드 작업이 되었나요, 아니면 상태만 DONE인가요?”
  * “마감이 다가오는데, 실제로 움직이는 흔적(커밋)이 있는지요?”
* 실제로는:
  * PM은 Task 상태만 보고 상황을 추정하거나,
  * 개발자에게 일일이 물어봐야 한다.
  * 설계된 Flow Chart와 실제 작업 사이의 연결이 약해서
    **“설계대로 구현이 진행되는지”를 체계적으로 보기 어렵다.**

이 서비스가 제공하는 해결책

1. **Flow Chart ↔ Task 연결**
   * 기획/PM이 만든 Flow(기능 흐름)를 시스템에 정의한다.
   * 각 단계(Flow Node)에 관련 Task들을 연결한다.
   * → 설계 관점에서 “이 단계는 어떤 Task로 구현되고 있는지”를 바로 본다.
2. **Task ↔ Commit 자동/수동 연결**
   * 개발자가 커밋 메시지에 Task 키를 포함하면 시스템이 자동으로 Task와 커밋을 연결한다.
   * 필요한 경우, Task 상세 화면에서 수동으로 Commit을 연결할 수도 있다.
   * → 계획(Task)과 실제 실행(Commit)이 직접 연결된다.
3. **Elasticsearch 기반 프로젝트 분석**
   * Task 상태 변경, Commit, Flow 단계별 활동 등을 ES에 이벤트로 쌓는다.
   * ES 집계를 통해:
     * 마감 임박 + 활동 없는 Task
     * Flow 단계별 완료율/지연 Task
     * 주간/월간 활동량/건강 점수
   * 같은 지표를 생성한다.
   * → PM은 “문제가 될 만한 영역”을 빠르게 찾을 수 있다.

1.5 서비스가 다루는 핵심 개념 (요약)

 1) 프로젝트(Project)

* 하나의 제품/서비스 단위.
* 하나 이상의 Git 저장소와 연결될 수 있다.
* Flow Chart, Task, Commit이 모두 프로젝트에 속한다.

2) Flow Chart / Flow Node

* **Flow Chart**
  * 기획/PM이 설계한 기능·서비스의 흐름.
  * 예: "회원가입 Flow", “주문/결제 Flow”
* **Flow Node**
  * Flow Chart 내의 개별 단계/박스.
  * 예: “이메일 입력”, “인증 메일 발송”, “결제 승인 요청”
* **이 서비스에서 Flow의 역할**
  * “기획/설계 뷰”의 기준이 되는 축
  * 각 Node에 관련 Task들을 연결해서,
    * 설계 단계마다 구현·진행 상황을 보는 기준으로 사용

3) Task

* 해야 할 작업/이슈 단위.
* 상태(TODO/IN\_PROGRESS/DONE), 담당자, 마감일 등을 가진다.
* 특정 Flow Node에 연결될 수 있다.
  * "이 Task는 이 Flow 단계 구현"이라는 의미.

4) Commit

* Git에서 실제 코드 변경 단위.
* 메시지, 작성자, 시간, 브랜치 등의 메타데이터를 가진다.
* 커밋 메시지에 Task 번호가 있으면 자동으로 Task와 연결된다.

5) 연결 관계

*Flow Node ↔ Task*
  * 설계된 흐름의 어느 단계에서 어떤 Task들이 나오는지 표현
*Task ↔ Commit*
  * 계획된 Task가 실제로 어떤 코드 변경으로 실행되었는지 표현
이 두 연결 덕분에:
> Flow(설계) → Task(계획) → Commit(실행)
이 세 단계의 흐름을 하나로 이어서 볼 수 있다.

6) 활동 데이터 & 분석 (Elasticsearch)

* Task 상태 변경, Commit, 기타 이벤트를
  &#x20;"프로젝트 활동 로그"로써 ES에 저장한다.
* ES에서:
  * 기간별 활동량
  * Flow 단계별/Task별 활동 패턴
  * 지연/정체 구간
* 등을 집계하고, 프로젝트·Flow별 평가 지표/점수를 만든다.

1.6 MVP에서 제공할 핵심 가치 (요약)

MVP 단계에서 이 서비스가 꼭 제공해야 하는 가치는 다음과 같다.
1. **Flow 기준으로 현재 구현 상태를 본다**
   * 프로젝트의 Flow Chart에서 각 단계별:
     * 연결된 Task 수
     * Task 상태(DONE 비율, IN\_PROGRESS 수 등)
   * 를 확인할 수 있다.
2. **Task 기준으로 실제 작업 여부를 확인한다**
   * Task 상세 화면에서:
     * 연결된 Commit 목록
     * 최근 Commit 시점
   * 을 확인해, 상태만 바뀐 Task와 실제로 작업이 진행된 Task를 구분할 수 있다.
3. **프로젝트/Flow별 리스크 포인트를 자동으로 보여준다**
   * 마감이 임박했는데 활동(Commit/상태 변경)이 없는 Task
   * 설계상 중요한 Flow Node인데 Task가 없거나 대부분 미진행인 곳
   * 최근 활동이 거의 없는 프로젝트/Flow
     이런 것들을 한 화면에서 모아서 보여준다.
4. **Elasticsearch를 활용한 분석 기반**
   * 활동 로그를 ES에 쌓고,
   * 프로젝트별/Flow별 기본 지표를 집계할 수 있는 구조를 만들어둔다.
   * 이후 단계에서:
     * 프로젝트 점수화
     * LLM 기반 평가/요약
   * 등으로 확장할 수 있는 기반이 된다.
