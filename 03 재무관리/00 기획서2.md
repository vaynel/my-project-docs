# 개인 자산 성장 시스템 기획서 (소비 관찰·업데이트 중심)

## 0. 한 줄 요약
카드 소비를 SMS 기반으로 자동 수집하고, 미분류/오류만 빠르게 수정하는 UI(Directus)를 중심으로 주간 리포트/경고를 돌려 **지출 통제 → 저축/투자 여력 확보 → 순자산 증가**를 지속시키는 시스템.

---

## 1. 배경 및 목표

### 1.1 현재 상황
- 수입원
  1) 월급
  2) 주식
  3) 기타
- 월급은 이미 일정(자동화)으로 여러 통장으로 분배됨
- 핵심 병목은 소비(카드 지출)의 “자동 수집 + 정확도 + 빠른 업데이트(수정)”임

### 1.2 목표
- 소비 데이터를 자동으로 수집하고(가능한 한 실시간), 오분류/누락을 빠르게 보정할 수 있는 운영 흐름 확보
- 주간 리포트로 소비 패턴을 계속 관찰하여 지출을 통제
- 장기적으로 저축률/투자비율 목표를 지키도록 경고/리포트 강화

---

## 2. 범위 정의

### 2.1 Phase 1 (필수)
- 카드 소비 SMS 기반 자동 수집(실시간/준실시간)
- 소비 데이터 표준화(거래 스키마 통합) 및 중복 방지 업서트
- 분류(카테고리) 규칙 기반 자동 분류 + 인박스(미분류) 처리 UI
- 주간 리포트 자동 전송(Discord)
- 월간 “정산 보정”을 위한 CSV 적재(선택이지만 권장)

### 2.2 Phase 2 (고도화)
- 구독/정기결제 자동 탐지
- 이상 지출 탐지(급증/빈도/패턴 변화)
- 분류 추천(규칙 + 유사도 추천)
- 주식/기타 수입 반영 고도화(스냅샷/이벤트 기반)

---

## 3. 핵심 원칙(운영·보안)

### 3.1 접속/노출 원칙
- 모든 웹 UI는 VPN 내부에서만 접근
- DB는 외부 포트 노출 금지(내부 컨테이너 네트워크만)
- Webhook 수신도 가능하면 VPN/내부망에서만 수신(불가 시 IP 제한/토큰 필수)

### 3.2 데이터 최소 수집
- SMS 원문 전체 저장은 최소화(필요 필드만 추출 저장)
- 개인식별 정보는 저장하지 않거나 마스킹

### 3.3 정합성 전략(현실적 이중화)
- SMS: 빠른 인지/실시간 반영(운영 편의)
- CSV: 취소/환불/할부/정산 반영(정확도)
- Phase 1에서는 “SMS로 운영 + 월 1회 CSV로 보정”을 기본으로 권장

---

## 4. 시스템 아키텍처

### 4.1 구성도
[휴대폰 SMS 수신]
    -> (SMS Forwarder 앱: HTTP POST)
    -> n8n Webhook
    -> PostgreSQL(Core DB)
    -> Directus(운영 UI: 업데이트/보정)
    -> Discord(주간/월간 리포트 및 경고)

### 4.2 구성 요소 역할
- n8n: 수집/파싱/정규화/업서트/스케줄 리포트
- PostgreSQL: 단일 진실 소스(Core DB), 집계/뷰/KPI
- Directus: “업데이트(수정) 중심” 운영 UI
- Discord: 리포트/경고 채널

---

## 5. 데이터 설계(Phase 1 최소 스키마)

### 5.1 계좌/수입원
- accounts
  - name: 월급통장, 소비통장, 증권계좌 등
  - type: bank | card | brokerage | cash
  - institution: 예) SHINHAN, KIWOOM 등(자유 입력)
  - is_active

- income_sources (선택)
  - name: 월급 | 주식 | 기타
  - description

### 5.2 거래(핵심)
- transactions
  - id
  - occurred_at: 승인/발생 시각(타임존 KST 기준 저장 권장)
  - amount: 양수로 저장
  - direction: income | expense | transfer
  - account_id: 발생 계좌(카드/통장)
  - merchant_raw: 가맹점 원문
  - merchant_norm: 정규화 가맹점(규칙/수정 반영)
  - memo
  - status: pending | confirmed | reversed (취소/환불 등)
  - external_id: 중복 방지 키(유니크)
  - source: sms | csv | manual
  - created_at, updated_at

중복 방지(external_id) 권장 생성 규칙:
- hash(occurred_at_분단위 + amount + merchant_raw + card_label(optional) + 승인/취소 구분)
- 카드사 문자 포맷에 따라 필드 조합 조정

### 5.3 카테고리(멀티)
- categories
  - id
  - name
  - parent_id (트리)
  - is_fixed_cost (고정비 여부)
  - budget_monthly (선택)

- transaction_categories (N:N)
  - transaction_id
  - category_id
  - weight (기본 1.0, 분할 필요 시 사용)

### 5.4 규칙 기반 자동 분류
- category_rules
  - id
  - pattern_type: contains | regex
  - pattern: 가맹점/메모 키워드
  - category_id
  - priority
  - is_active
  - exclude_pattern (선택)

## 6. 수집 및 처리 로직

### 6.1 카드 소비 SMS 수집(Phase 1 핵심)
1) SMS Forwarder 앱이 SMS 수신 시 n8n Webhook으로 POST
2) n8n에서 파싱
   - 승인/취소 구분
   - 금액
   - 가맹점
   - 발생 시각
3) 표준화
   - occurred_at, amount, merchant_raw 등 필드 매핑
4) external_id 생성 후 upsert
5) 자동 분류 룰 적용
6) 미분류/의심 거래는 “인박스”로 남김

참고: SMS Forwarder는 오픈소스 기반을 사용해도 되고, 상용 포워더를 써도 됨(원칙은 동일)

### 6.2 CSV 정산 보정(권장)
- 월 1회 또는 주 1회 카드 이용내역 CSV를 수동 다운로드 후 지정 폴더에 저장
- n8n이 파일을 읽어:
  - 취소/환불/할부 정산 반영
  - SMS에서 들어온 pending 거래를 confirmed로 전환
  - 동일 거래 매칭 로직으로 상태 갱신(occurred_at/amount/merchant 기준)

---

## 7. 운영 UI(Directus) 설계

### 7.1 인박스 뷰(핵심 화면)
- 필터 조건 예시
  - 카테고리 미지정
  - merchant_norm 미지정
  - status = pending
  - 이상치 플래그(Phase 2)

### 7.2 빠른 업데이트 기능
- 거래 클릭 시 즉시 편집:
  - 카테고리(멀티) 지정
  - merchant_norm 수정
  - 고정비 체크
  - 메모 입력
  - “규칙으로 저장” 액션(다음부터 자동 분류)

### 7.3 규칙 관리 화면
- category_rules CRUD
- 우선순위 관리(충돌 시 높은 priority 우선)
- 테스트(특정 가맹점 문자열로 룰 미리보기) 기능(가능하면)

---

## 8. KPI 및 리포트(소비 통제 중심)

### 8.1 Phase 1 KPI
- 주간 총지출
- 카테고리 Top 3 지출
- 전주 대비 증감(%) 및 금액
- 미처리(미분류/보정 필요) 거래 수
- 고정비 비율(고정비/총지출)

### 8.2 Phase 2 KPI(확장)
- 저축률: (수입 - 지출)/수입
- 투자비율: 투자 관련 이동/매수 금액 비중 또는 투자자산/총자산
- 이상 지출 탐지: 최근 4주 평균 대비 급증, 특정 카테고리 급증 등

### 8.3 Discord 리포트 템플릿(권장)
- 주간(매주 월요일 오전):
  - 이번 주 총지출
  - Top3 카테고리
  - 전주 대비
  - 미처리 건수
  - 경고(예산 초과/급증)
- 월간(매월 1일 오전):
  - 월 총지출
  - 고정비 비율
  - 목표 대비 상태(설정값 기반)

---

## 9. 자동화(Workflow) 상세 설계

### 9.1 n8n 워크플로우 목록
1) sms_ingest_webhook
   - Webhook -> Parse -> Normalize -> Upsert(transactions) -> Apply rules -> Notify(실패 시)

2) csv_reconcile_import
   - Trigger(수동/스케줄/폴더 감시)
   - Parse CSV -> Normalize -> Match existing -> Update status/amount/merchant -> Apply rules -> Summary log

3) weekly_report
   - Cron(매주 월요일)
   - SQL query -> Format -> Discord webhook

4) monthly_report
   - Cron(매월 1일)
   - SQL query -> Format -> Discord webhook

5) data_quality_check (선택)
   - 누락/중복/이상치 검출 후 Discord 경고

## 10. 구현 계획(실행 순서)

### 10.1 Phase 0: 인프라(1~2일)
- Docker Compose: PostgreSQL + Directus + n8n
- VPN 내부 접근만 허용(방화벽/리버스프록시 정책)
- 백업: PostgreSQL daily dump + 7~30일 보관

산출물:
- docker-compose.yml
- .env 템플릿
- 백업 스크립트/크론

### 10.2 Phase 1: 소비 수집/업데이트(3~7일)
- 스키마 생성
- Directus 컬렉션/권한 구성
- SMS 포워딩 -> n8n Webhook ingest 구현
- category_rules 적용
- 인박스 뷰 구성
- 주간 리포트 구현

산출물:
- schema.sql
- n8n 워크플로우 export
- Discord 리포트 템플릿

### 10.3 Phase 1.5: 정산 보정(선택, 2~5일)
- 카드 이용내역 CSV 파서/매칭 로직
- pending -> confirmed 전환 로직
- 취소/환불 처리

산출물:
- csv_reconcile 워크플로우
- 매칭 규칙 문서

### 10.4 Phase 2: 고도화(2~4주)
- 구독 탐지, 이상 지출 탐지
- 추천 분류(유사도/빈도 기반)
- 저축률/투자비율 KPI 확장

## 11. 운영 루틴(지속성 확보)

- 매일: 인박스 미분류 처리 1~3분
- 매주: 주간 리포트 확인 + 예산/규칙 보정 10분
- 매월: CSV 정산 보정 10~20분(가능하면)

---

## 12. 리스크 및 대응

- SMS 포맷 변경
  - 파서 정규식/맵핑을 설정화하고, 실패 시 즉시 알림
- 중복 적재/누락
  - external_id 유니크 + 업서트
  - 데이터 품질 체크 워크플로우
- 취소/환불/할부 정산 정확도
  - CSV 보정 루틴 도입(월 1회라도 권장)
- 보안
  - VPN-only, 토큰 기반 Webhook, DB 외부 노출 금지, 정기 업데이트, 백업

---

## 13. 결정해야 할 항목(설계 확정 체크리스트)

1) 신한카드 결제 알림 형태
- SMS로 오는지
- 푸시만 오는지(이 경우 수집 난이도 상승)

2) 카드 승인 문자 샘플 2~3개(개인정보 마스킹)
- 파싱 규칙 확정용

3) CSV 정산 보정 주기
- 월 1회(권장) / 주 1회 / 안 함
