### S-01. IDOR ⚠️ 부분 구현

**상태:** 미완료 (보안 명세서 기준)

**확인 내용:**
- `report-files.service.ts`: `findAll(schoolId?)` 에서 `schoolId` 파라미터로만 필터링
- 요청한 사용자가 해당 `schoolId`에 속하는지 **소유권 검증 코드 없음**
- `ForbiddenException` 처리 부재 — role이 `user`인 경우에도 타 학교 ID를 직접 입력하면 접근 가능

**필요 조치:**
- `report-files.service.ts`의 `findOne`, `findAll`에서 `user.role !== 'admin'`일 때 `user.schoolId === record.schoolId` 검증 추가
- 보안 명세서 S-01 패치 코드 참조

### S-03. 파일 업로드 취약점 ⚠️ 부분 구현

**상태:** 핵심 검증은 구현, 사이즈 제한 불일치

**확인 내용 (구현됨):**
- MIME 타입 화이트리스트: ✅ (`FileSecurityValidator.ALLOWED_MIME_TYPES`)
- Magic Bytes 검증: ✅ (`MAGIC_NUMBERS` 기반 헤더 비교)
- 악성 코드 패턴 스캔: ✅ (`<script>`, `eval(`, `system(` 등)
- 실행 권한 제거: ✅ (`removeExecutePermissions`)
- UUID 기반 파일명 생성: ✅ (`generateUniqueFileName`)

**문제점:**
- 파일 크기 제한이 **500MB** — 명세서 기준 50MB의 10배
- `main.ts`의 body parser도 `limit: '500mb'` — DoS 위험
- `file-type` 라이브러리 미사용 (자체 Magic Bytes 구현, 커버리지 제한적)

**필요 조치:**
```typescript
// file-security.util.ts
const maxSize = 50 * 1024 * 1024; // 500MB → 50MB

// main.ts
app.use(express.json({ limit: '50mb' }));
app.use(express.urlencoded({ limit: '50mb', extended: true }));
```

### S-05. XSS ⚠️ 부분 구현

**상태:** 입력 검증은 있으나 HTML sanitize 미적용

**확인 내용:**
- `main.ts`: `ValidationPipe({ whitelist: true })` — DTO 외 필드 제거 ✅
- `forbidNonWhitelisted: false` — 허용되지 않은 필드가 **경고 없이 통과** (차단 아님)
- DTO에 `@IsString()`, `@MaxLength()` 데코레이터는 있으나 HTML 태그 제거 Sanitize 없음
- `security-headers.middleware.ts`의 CSP에 `'unsafe-inline'` + `'unsafe-eval'` 포함 → CSP의 XSS 방어력 약화

**필요 조치:**
- `forbidNonWhitelisted: true`로 변경
- DTOs에 HTML 태그 sanitize 적용 (`class-sanitizer` 또는 자체 pipe)
- CSP에서 `'unsafe-eval'` 제거 (프론트 코드 수정 필요할 수 있음)

### S-07. HTTP 보안 헤더 ⚠️ 부분 구현

**상태:** 미들웨어 적용됨, CSP 강도 약화

**확인 내용 (구현됨):**
- `SecurityHeadersMiddleware` 전역 적용 (`app.module.ts`) ✅
- HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy ✅
- Permissions-Policy, COEP, COOP, CORP 추가 헤더 ✅

**문제점:**
```
Content-Security-Policy: script-src 'self' 'unsafe-inline' 'unsafe-eval'
```
- `'unsafe-inline'`: 인라인 스크립트 허용 → XSS 방어 무력화
- `'unsafe-eval'`: `eval()` 허용 → 코드 인젝션 위험

**필요 조치:**
- `'unsafe-eval'` 제거 (프론트엔드 동적 코드 평가 여부 확인 후)
- `'unsafe-inline'` 제거 후 nonce 방식으로 전환 (장기 목표)
