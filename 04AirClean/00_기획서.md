### 1. 개요 (Introduction)

- **프로젝트 배경:** 아버지 회사 홈페이지 제작
    
- **서비스 목표:** 
	- 랜딩 페이지 
	- 보고서 관리
	
- **주요 용어 정의**	

| 분류     | 엔티티명        | 설명 (Description) | 핵심 속성 (Key Attributes)                              |
| ------ | ----------- | ---------------- | --------------------------------------------------- |
| user   | user        | 서비스 사용자          | id, username, password, school_id, role, created_at |
|        | admin       | 서비스 관리자<br>(아버지) | id, admin_id, password, role, last_login            |
| report | report      | 보고서              | id, title, content, user_id, status, created_at     |
|        | report_file | 보고서 파일           | id, report_id, file_url, file_size, uploaded_at     |
|        | school      | 사용자의 학교          | id, school_name, region, contact_number             |

### 2. 요구사항 정의 (Requirements)

- **기능 요구사항 (Functional)**
    - user 관리 (회원가입, 로그인, 권한 분리: admin/user)
    - user 별 보고서 관리 (관리자의 보고서 등록 및 유저의 보고서 조회/다운로드)
    - user(school)별 담당자 관리 (학교별 다중 사용자 매핑)
    - 랜딩 페이지 기능 (회사 소개, 서비스 안내, 고객 문의 폼)
        
- **비기능 요구사항 (Non-functional)**
    - **보안:** 사용자 비밀번호 단방향 암호화(SHA-256 + Salt 또는 PBKDF2 등 KISA 권고 알고리즘 적용), JWT 기반 API 인증 적용
    - **가용성:** 정기 데이터베이스 백업 및 장애 발생 시 데이터 복구 체계 마련

### 3. 개발 목표 및 방향 (Development Objectives)

- **기술적 지표**
	- 조회 속도 100ms 미만
	- 대용량 파일 다운로드 시 API 병목 현상 최소화
    
- **핵심 도전 과제**
	- 백업 시스템 구축 (DB 데이터 및 스토리지 파일)
	- 대용량 보고서 파일(PDF, 이미지 등)의 로컬(Local) 디스크 저장 및 백업 서버에 Dump 방식으로 주기적 연동
	- 사용자 권한(Admin vs User)에 따른 엄격한 데이터 접근 제어
    

### 4. 시스템 아키텍처 및 기술 스택 (System Architecture)

| ----- | ------------ |
| 프론트   | Next.js      |
| 백엔드   | Nest.js      |
| 언어    | node.js 22.0 |
| DB    | MariaDB      |


### 5. 상세 구현 계획 (Implementation Plan)

- **모듈별 상세 설계:** 컴포넌트 구조, 핵심 알고리즘 Flowchart
    - Next.js 기반 랜딩 페이지(SSG)와 대시보드(CSR/SSR) 컴포넌트 분리 설계
    
- **데이터 흐름:** 클라이언트와 서버 간의 상호작용 flow
    - RESTful API 통신 기준, 인증 토큰(JWT)을 포함한 요청 및 응답 구조 설계
    
- **에러 핸들링:** 예외 상황 정의 및 처리 전략
    - Nest.js Global Exception Filter를 활용한 통합 에러 로깅 및 공통 에러 응답 포맷 구성
