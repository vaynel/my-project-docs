### S-01. IDOR (Insecure Direct Object Reference)

**위험도:** 🔴 Critical

**설명**
`GET /reports/10` 요청 시 user가 자신의 school에 속하지 않은 report_id를 직접 입력해 타 학교 보고서에 접근 가능.

**취약 시나리오**
```
user (school_id: 1)가 GET /reports/99 요청
→ school_id 검증 없이 응답 시 타 학교 보고서 노출
```

**패치 방법 (Nest.js Service Layer)**
```typescript
// reports.service.ts
async findOne(reportId: number, user: JwtPayload) {
  const report = await this.reportRepo.findOne({ where: { id: reportId } });

  if (!report) throw new NotFoundException();

  // user 역할이 admin이 아니면 school_id 반드시 검증
  if (user.role !== 'admin' && report.school_id !== user.school_id) {
    throw new ForbiddenException();
  }

  return report;
}
```

**체크리스트**
- [ ] Reports 단건 조회에 school_id 검증 추가
- [ ] Report Files 조회/다운로드에 school_id 검증 추가
- [ ] Schools 단건 조회에 본인 소속 학교 여부 검증 추가

### S-02. Path Traversal (파일 경로 탐색)

**위험도:** 🔴 Critical

**설명**
파일 저장 경로가 `{FILE_STORAGE_ROOT}/{school_name}/{date}/{filename}` 구조이므로, school_name이나 filename에 `../`가 포함되면 루트 디렉토리까지 탐색 가능.

**취약 시나리오**
```
school_name: "../../etc"
filename: "passwd"
→ 저장 경로: /var/data/reports/../../etc/passwd
```

**패치 방법**
```typescript
import * as path from 'path';

function sanitizeDirName(name: string): string {
  // 한글, 영문, 숫자, 하이픈, 언더스코어만 허용
  return name.replace(/[^a-zA-Z0-9가-힣_-]/g, '_');
}

function buildSafePath(root: string, schoolName: string, date: string, filename: string): string {
  const safeName = sanitizeDirName(schoolName);
  const safeDate = date.replace(/[^0-9-]/g, '');  // YYYY-MM-DD만 허용
  const safeFile = path.basename(filename);        // 디렉토리 부분 제거

  const fullPath = path.join(root, safeName, safeDate, safeFile);

  // 최종 경로가 반드시 root 하위인지 검증
  if (!fullPath.startsWith(path.resolve(root))) {
    throw new BadRequestException('Invalid file path');
  }

  return fullPath;
}
```

**체크리스트**
- [ ] school_name 디렉토리명 sanitize 함수 적용
- [ ] filename에 `path.basename()` 적용 (디렉토리 제거)
- [ ] 최종 경로가 FILE_STORAGE_ROOT 하위인지 `startsWith` 검증
- [ ] date 형식 YYYY-MM-DD 정규식 검증

### S-03. 파일 업로드 취약점

**위험도:** 🔴 Critical

**설명**
확장자 검증 없이 파일을 수락하면 `.php`, `.js`, `.sh` 등 실행 파일 업로드 후 서버 실행 가능(WebShell 공격).

**패치 방법**
```typescript
// file-upload.interceptor.ts
const ALLOWED_MIME_TYPES = [
  'application/pdf',
  'image/jpeg',
  'image/png',
  'image/webp',
  'application/vnd.openxmlformats-officedocument.wordprocessingml.document', // docx
  'application/vnd.openxmlformats-officedocument.spreadsheetml.sheet',       // xlsx
];

const MAX_FILE_SIZE = 50 * 1024 * 1024; // 50MB

// Multer 설정
export const multerOptions: MulterOptions = {
  limits: { fileSize: MAX_FILE_SIZE },
  fileFilter: (req, file, cb) => {
    if (!ALLOWED_MIME_TYPES.includes(file.mimetype)) {
      return cb(new BadRequestException('허용되지 않는 파일 형식입니다.'), false);
    }
    cb(null, true);
  },
};
```

> **주의:** MIME 타입은 클라이언트가 조작 가능. 실제 파일 시그니처(Magic Bytes)를 서버에서 검증하는 `file-type` 라이브러리 추가 권장.

```bash
npm install file-type
```

**SHA-256 해시 저장 (무결성 방어층)**
```typescript
import { createHash } from 'crypto';
import { createReadStream } from 'fs';

async function computeFileHash(filePath: string): Promise<string> {
  return new Promise((resolve, reject) => {
    const hash = createHash('sha256');
    createReadStream(filePath)
      .on('data', (chunk) => hash.update(chunk))
      .on('end', () => resolve(hash.digest('hex')))
      .on('error', reject);
  });
}

// 업로드 직후 호출 → report_file.file_hash 컬럼에 저장
const fileHash = await computeFileHash(savedFilePath);
```

> 다운로드 응답 헤더에 `X-File-Hash: sha256:{hash}` 포함. 클라이언트는 수신 파일을 재해싱하여 비교.
> `GET /reports/:reportId/files/:fileId/verify`로 서버 측 재해싱 검증 가능 (백업 복원 후 무결성 확인 용도).

**체크리스트**
- [ ] MIME 타입 화이트리스트 적용
- [ ] 파일 크기 제한 설정 (50MB)
- [ ] `file-type` 라이브러리로 Magic Bytes 검증
- [ ] 업로드 파일명에 UUID 접두사 부여 (원본 파일명 노출 방지)
- [ ] Nginx에서 업로드 디렉토리 실행 권한 제거 (`location /files/ { ... }`)
- [ ] 업로드 완료 직후 SHA-256 해시 계산 → `report_file.file_hash` 저장
- [ ] 다운로드 응답 헤더에 `X-File-Hash` 포함
- [ ] `/verify` 엔드포인트에서 디스크 파일 재해싱 → DB 해시 비교 구현

### S-04. SQL Injection

**위험도:** 🟠 High

**설명**
Raw Query 사용 시 사용자 입력이 SQL 구문으로 해석될 수 있음.

**패치 방법**
```typescript
// ❌ 위험: Raw Query에 직접 삽입
const result = await dataSource.query(
  `SELECT * FROM report WHERE title = '${userInput}'`
);

// ✅ 안전: TypeORM Parameterized Query
const result = await reportRepo.findOne({
  where: { title: userInput }
});

// ✅ 안전: QueryBuilder 사용 시 파라미터 바인딩
const result = await reportRepo
  .createQueryBuilder('r')
  .where('r.title = :title', { title: userInput })
  .getOne();
```

**체크리스트**
- [ ] Raw Query 전수 검사 → Parameterized Query로 교체
- [ ] TypeORM 사용 원칙 팀 공유
- [ ] Nest.js ValidationPipe 전역 설정으로 DTO 타입 강제

### S-05. XSS (Cross-Site Scripting)

**위험도:** 🟠 High

**설명**
보고서 `title`, `content` 등에 `<script>` 태그 입력 시 다른 사용자 브라우저에서 실행 가능.

**패치 방법 (백엔드 - Nest.js)**
```bash
npm install class-sanitizer
```

```typescript
// main.ts - 전역 ValidationPipe 설정
app.useGlobalPipes(new ValidationPipe({
  whitelist: true,        // DTO에 없는 속성 자동 제거
  forbidNonWhitelisted: true,
  transform: true,
}));
```

```typescript
// create-report.dto.ts
import { IsString, MaxLength } from 'class-validator';
import { Sanitize } from 'class-sanitizer';

export class CreateReportDto {
  @IsString()
  @MaxLength(200)
  @Sanitize(e => e.replace(/<[^>]*>/g, ''))  // HTML 태그 제거
  title: string;
}
```

**패치 방법 (프론트엔드 - Next.js)**
```typescript
// 사용자 입력을 HTML로 렌더링할 때 DOMPurify 사용
import DOMPurify from 'dompurify';

const safeContent = DOMPurify.sanitize(report.content);
```

**체크리스트**
- [ ] 전역 ValidationPipe 설정 (`whitelist: true`)
- [ ] DTO에 `@IsString()`, `@MaxLength()` 데코레이터 적용
- [ ] 프론트엔드 HTML 렌더링 시 DOMPurify 적용
- [ ] Content-Security-Policy 헤더 설정 (S-07 참조)

### S-07. HTTP 보안 헤더 미설정

**위험도:** 🟡 Medium

**패치 방법 (Nginx)**
```nginx
# /etc/nginx/conf.d/security-headers.conf

add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Content-Security-Policy "
  default-src 'self';
  script-src 'self';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data:;
  font-src 'self';
  connect-src 'self';
  frame-ancestors 'none';
" always;
```

| 헤더 | 목적 |
|---|---|
| `HSTS` | HTTP → HTTPS 강제 리다이렉트 |
| `X-Frame-Options: DENY` | 클릭재킹(iFrame 삽입) 방어 |
| `X-Content-Type-Options: nosniff` | MIME 스니핑 방어 |
| `Content-Security-Policy` | 인라인 스크립트, 외부 리소스 로딩 차단 |
| `Referrer-Policy` | Referrer 헤더 정보 노출 제한 |

**체크리스트**
- [ ] Nginx 보안 헤더 설정 파일 추가
- [ ] HSTS 적용 (HTTPS 강제)
- [ ] CSP 정책 도메인에 맞게 조정

**패치 방법**
```typescript
// logger.interceptor.ts - 민감 필드 마스킹
const SENSITIVE_FIELDS = ['password', 'accessToken', 'refreshToken', 'otpCode', 'sessionToken'];

function maskSensitiveData(obj: Record<string, unknown>): Record<string, unknown> {
  return Object.fromEntries(
    Object.entries(obj).map(([key, value]) =>
      SENSITIVE_FIELDS.includes(key) ? [key, '***MASKED***'] : [key, value]
    )
  );
}
```

### S-11. 에러 메시지 정보 노출

**위험도:** 🟡 Medium

**설명**
Stack trace, 파일 경로, DB 쿼리 정보가 클라이언트에 노출되면 공격자에게 내부 구조 파악 기회 제공.

**패치 방법 (Nest.js Global Exception Filter)**
```typescript
// all-exceptions.filter.ts
@Catch()
export class AllExceptionsFilter implements ExceptionFilter {
  private readonly logger = new Logger(AllExceptionsFilter.name);

  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();

    const status = exception instanceof HttpException
      ? exception.getStatus()
      : HttpStatus.INTERNAL_SERVER_ERROR;

    // 서버 에러는 내부 로그에만 기록, 클라이언트에는 일반 메시지만 반환
    if (status >= 500) {
      this.logger.error(exception);  // 내부 로그
      return response.status(status).json({
        success: false,
        statusCode: status,
        message: '서버 오류가 발생했습니다.',  // 상세 정보 미노출
      });
    }

    // 4xx 에러는 메시지 전달
    const message = exception instanceof HttpException
      ? exception.message
      : '요청을 처리할 수 없습니다.';

    response.status(status).json({ success: false, statusCode: status, message });
  }
}
```

**체크리스트**
- [ ] Global Exception Filter 등록 (`main.ts`)
- [ ] 500 에러 시 클라이언트 응답에서 stack trace 제거
- [ ] `NODE_ENV=production` 환경에서 상세 에러 비활성화
